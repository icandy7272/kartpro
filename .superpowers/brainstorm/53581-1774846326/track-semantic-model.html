第二段设计: 赛道语义骨架层的数据模型

目标: 让后续所有教练建议都引用同一套赛道真相, 并且能记录自动推断、置信度和人工确认。

核心对象
TrackSemanticModel
trackId
version
sourceLapId
corners[]
straights[]
relationships[]
semanticTags[]
pendingConfirmations[]
CornerSemantic
cornerId / name
start / apex / end
angle / direction / type
followingStraightLength
importanceScore
manualStatus: auto | confirmed | overridden
SemanticTag
tagType
targets: [T3] / [T5,T6]
confidence
reasonCodes
explanation
manualDecision
第一阶段 tag 类型
must-hit-exit
关键出弯弯 / 长直道入口弯

compound-corner
组合弯候选

sacrifice-entry
牺牲当前弯, 服务后续弯或直道

setup-corner
铺垫弯 / 关键准备弯
置信度与人工确认回写
高置信度
直接进入教练分析
但仍保留 reasonCodes 和 confidence
中置信度
作为候选展示
支持一键确认
或改判
低置信度
不进入强结论
仅作为提示
等待人工确认
人工确认不是直接改原始几何，而是写回:
manualDecision = confirmed | rejected | overridden
这样后续教练建议能知道: 这个语义是系统猜的，还是用户确认过的。
A

模型方向对

对象拆分和人工确认回写方式合理，可以继续讲交互和流程。

B

模型基本对，但字段或边界要改

整体没问题，但对象粒度、tag 类型或 manualStatus 设计需要调整。

C

模型太重了

第一阶段不该引入这么完整的骨架层，想先做更轻的版本。
